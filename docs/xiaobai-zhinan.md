## 一、这个项目在做什么？

一句话：**假装你是一个刚入职的数据分析师，老板给了你一份用户行为数据，让你搞清楚用户是怎么用的、哪里有问题、怎么改进。**

你做出来的东西：
- 5 个 Jupyter Notebook（分析脚本 + 图表 + 文字结论）
- 1 个 GitHub 仓库（项目的主页，面试官点开链接就能看到你做了什么）

面试官打开你的 GitHub 链接后看到的是：
1. 一个 README（项目说明书，告诉别人这是什么的）
2. 5 个 Notebook（每个里面有代码、图表、你的分析结论）
3. 12 张图表（转化漏斗、留存曲线、用户分层饼图……）

## 二、每个文件的后缀是什么意思？

| 后缀 | 全称 | 白话解释 |
|------|------|---------|
| `.md` | Markdown | 带格式的纯文本文件。`#` 是标题，`-` 是列表，`**粗体**` 是加粗。GitHub 自动把它渲染成漂亮的网页 |
| `.py` | Python | Python 代码文件，可以被 `python xxx.py` 运行 |
| `.ipynb` | IPython Notebook | Jupyter Notebook 的格式。包含代码块 + 输出结果（图表/表格）+ 文字说明，三合一 |
| `.csv` | Comma-Separated Values | 表格数据文件，每一行是一条记录，每一列用逗号隔开。用 Excel 可以直接打开 |
| `.db` | Database | SQLite 数据库文件，里面存着表格，可以用 SQL 查询 |
| `.gitignore` | — | 告诉 Git「这些文件不要管」。比如数据文件太大，不上传 |
| `.txt` | Text | 纯文本文件，没有任何格式 |
| `.png` | Portable Network Graphics | 图片文件 |
| `.gitkeep` | — | 一个约定俗成的空文件，用来让 Git 追踪一个空文件夹（Git 默认忽略空文件夹） |

## 三、每一步的用意是什么？

我们做了两件大事：**A) 搭项目骨架**，**B) 跑完 6 步分析**。

### A. 搭项目骨架

| 步骤 | 做了什么 | 为什么 |
|------|---------|--------|
| 创建文件夹结构 | `data/` `notebooks/` `utils/` `resume/` | 把代码、数据、工具、简历分开，面试官一看就知道你很规范 |
| 写 `.gitignore` | 列出不上传的文件 | 数据文件（`.csv` `.db`）太大，不应该进 GitHub；但图表 `.png` 需要上传展示 |
| 写 `requirements.txt` | 列出依赖的 Python 包 | 别人或未来的你自己，运行 `pip install -r requirements.txt` 就能一键装好所有包 |
| 写 `utils/db.py` | 共用的数据库工具函数 | 避免每个 Notebook 里重复写同样的代码（DRY 原则：Don't Repeat Yourself） |
| 写 `utils/generate_data.py` | 生成模拟数据 | 原数据集下载不了，我们按真实分布参数自己生成一份 |

### B. 6 步分析（每个 Notebook 一步）

| Notebook | 分析什么 | 回答什么问题 | 跟下一步的关系 |
|----------|---------|-------------|--------------|
| `01_data_cleaning` | 加载数据 → 查缺失值 → 查重复 → 时间字段标准化 | **数据可信吗？** | 后面所有分析都依赖这一步。脏数据进去，脏结论出来（Garbage In, Garbage Out） |
| `02_user_profile` | PV/UV、行为分布饼图、DAU 趋势、时段热力图、用户活跃度长尾 | **用户什么样？什么时候活跃？** | 为漏斗分析和留存分析提供基线。比如发现周末用户 50% 更多 → 漏斗也应该分工作日/周末看 |
| `03_conversion_funnel` | 浏览→收藏→加购→购买的每一步流失率，分品类对比，用户实际走了哪些路径 | **用户怎么买东西？哪里卡住了？** | 漏斗告诉你「哪里流失最多」→ 留存告诉你「流失的人回来吗」 |
| `04_retention_path` | 次日/3日/7日留存率，队列留存热力图，首次浏览→首次购买的时间间隔 | **用户会回来吗？多久下单？** | 留存告诉你「谁留住了」→ 用户分层告诉你「留住的人是谁」 |
| `05_user_segmentation` | 用 RFM 思路给每个用户打分，分成 5 层，分群对比行为 | **谁是好用户？怎么差异化运营？** | 分层结果直接导向运营建议：高价值用户给 VIP，流失风险用户给召回券 |
| README 里的「业务建议」 | 基于上面所有分析，给出 3 条可以立刻落地的建议 | **如果我是运营负责人，我做什么？** | 这是整个项目的「收束」。没有建议的分析就是自嗨 |

**关联链**：数据清洗 → 用户画像（知道整体情况）→ 转化漏斗（找到问题点）→ 留存分析（问题用户在离开）→ 用户分层（对不同类型的用户用不同策略）→ 业务建议

## 四、Git 是什么？

**Git 是一个版本控制工具。** 它记录你对文件做的每一次修改，让你可以：

- 回到过去的任何一个版本（"我三天前删了一段代码，现在想要回来"）
- 看到每次修改了什么、谁改的、什么时候改的
- 和另一个人同时编辑同一个项目而不会互相覆盖

通俗比喻：Git 就像是 Word 的「修订模式」+「另存为」的功能，但是是自动的，而且可以存无限个版本。

**关键概念**：

| 术语 | 白话 |
|------|------|
| `commit` | 保存一个快照。"我现在觉得这个状态不错，记下来" |
| `push` | 把你电脑上的 commit 上传到 GitHub（让别人也能看到） |
| `pull` | 把别人上传的 commit 下载到你的电脑 |
| `remote` | 远程仓库。就是一个远端地址，通常是 GitHub 的链接 |
| `branch` | 分支。你可以开一个新分支做实验，搞砸了切回主分支就行 |

## 五、GitHub 是什么？Git 和 GitHub 的关系？

- **Git** = 你电脑上的工具（免费，命令行操作）。属于你自己。
- **GitHub** = 微软旗下的一个网站。用来**存放** Git 仓库的。别人可以在浏览器里看到你的代码。

类比：
- Git = 手机相册 App（拍照、编辑、整理照片）
- GitHub = 微信朋友圈（把照片发出去给别人看）

**「Repo（仓库）」** = 一个项目的 Git + GitHub 存储空间。你的项目就是一个 repo，地址是：

```
https://github.com/BenjaminJ1899/taobao-user-behavior-analysis
```

这个 URL 就是你的「仓库地址」。面试官点开就能看到你的项目主页。

## 六、README 和 .gitignore 有什么用？

### README.md

GitHub 仓库的门面。当你打开一个 GitHub 链接，自动显示在文件列表下方的就是 README。

- 告诉别人这个项目是干嘛的
- 怎么用（怎么安装、怎么运行）
- 发现了什么（关键结论）
- 用了什么技术

没有 README 的仓库 ≈ 没有封面的书。面试官打开一个没有 README 的项目，3 秒就关了。

### .gitignore

告诉 Git **哪些文件不要管、不要上传**。

里面写的内容像这样：
```
# Data files
data/*.csv          ← 所有 CSV 文件不上传（太大）
data/*.db            ← 数据库文件不上传

# Python
__pycache__/         ← Python 自动生成的缓存文件夹不上传
```

为什么需要它？因为你的 CSV 数据文件可能有几百 MB，GitHub 有大小限制，而且面试官也不需要下载你的数据——他们只想看你的分析代码和结论。数据可以自己用脚本生成。

## 七、「push」是什么意思？

整个流程是这样的：

```
你在本地写代码  →  git add（标记哪些文件要保存）
                 →  git commit（在本机保存一个快照）
                 →  git push（把这个快照上传到 GitHub）
```

所以「我帮你 push」= 我帮你把本地的代码上传到 GitHub 上，让面试官能看到。

## 八、SSH Key 和 Personal Access Token 是什么？

这两个东西解决同一个问题：**证明「你是你」**。

GitHub 需要确认上传代码的人就是仓库的主人，不能随便什么人都能往你的仓库里推代码。

### SSH Key

你电脑上的一个加密文件（一对密钥：公钥 + 私钥）。
- 公钥放在 GitHub 上
- 私钥留你电脑上
- 每次 push，GitHub 用公钥验证你手里的私钥

类比：公钥是锁，私钥是钥匙。GitHub 上有你的锁，你每次操作带上钥匙（自动），就通过了。

### Personal Access Token (PAT)

一串长密码。当你用 HTTPS 链接 push 时（就是我们的 `https://github.com/BenjaminJ1899/...`），GitHub 会弹出一个框让你输入用户名和密码。密码框里填的不是你的 GitHub 密码，而是这个 PAT。

类比：QQ 的设备密码。你在一个不信任的设备上登录 QQ 时，用临时密码而不是主密码。

### Windows Git Credential Manager

我们刚才配了这个东西。它是 Windows 自带的一个工具，负责帮你记住 GitHub 密码/PAT。你登录一次之后就不再需要输入。

## 十、你现在打开 Notebook 要怎么看？

在 VS Code 中打开任意 `.ipynb` 文件：
1. 每个灰色方块是代码块（可以点播放按钮 ▶ 执行）
2. 每个白色区域是文字说明（Markdown，解释这一步在做什么）
3. 代码块的下面会显示输出结果（表格、数字、图表）
4. **按顺序从第一个代码块点 ▶ 开始执行**，不要跳着执行（后面的代码依赖前面的变量）

如果你的 VS Code 没有自动安装 Jupyter 支持，它会提示你安装。同意就行。
